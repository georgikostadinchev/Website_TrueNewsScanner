URL PIPELINE
File:
- artifacts/api-server/src/lib/pipeline/url.ts

Implementation goals:
- Move away from relying only on regex over the full input string
- Parse the URL and analyze components separately where possible:
  - protocol
  - hostname
  - pathname
  - query/search params
  - port
  - subdomain depth

Suggested helpers:
- tryParseUrl(input)
- normalizeHostname(hostname)
- getSubdomainDepth(hostname)
- hasRandomLookingPathSegment(pathname)
- hasRedirectParams(searchParams)
- isTrustedInstitutionDomain(hostname)
- isTrustedMediaDomain(hostname)

Risk signals to add:
1. IP address as hostname
   - hostname matches IPv4
   - weight: 0.65
   - Bulgarian label/description

2. BG bank brand typosquat / imitation
   - domain contains variants of:
     - unicredit
     - fibank
     - dsk-bank
     - postbank
     - raiffeisen
   - but is NOT an exact trusted domain
   - weight: 0.75

3. Excessive subdomain depth
   - more than 3 subdomains
   - weight: 0.5

4. Non-standard port
   - explicit port other than 80/443
   - weight: 0.45

5. Long random-looking path segment
   - path segment > 60 chars and looks machine-generated
   - weight: 0.35

6. Redirect chain indicators
   - multiple redirect=, url=, next=, goto= params
   - weight: 0.4

7. Mixed TLD squatting
   - trusted-looking brand/domain followed by extra suffix
   - example: bta.bg.info
   - weight: 0.55

8. No TLD / malformed URL
   - cannot be parsed properly or obviously malformed
   - weight: 0.3

9. Dangerous scheme
   - data: or javascript:
   - weight: 0.9

10. Expanded phishing TLD list
   - include .top, .click, .gq, .pw, .cc
   - weight: 0.4

Safe signals to add:
1. Known Bulgarian institution domain
   - exact allowlist match only
   - examples: bnb.bg, mvr.bg, and other official Bulgarian institution domains
   - weight: -0.4

2. Long-established Bulgarian media
   - exact match only:
     - bnt.bg
     - nova.bg
     - btv.bg
     - dir.bg
     - mediapool.bg
   - weight: -0.25

Important false-positive rule:
- Remove or narrow any overly broad “.bg means safe” logic
- Exact trusted matches must suppress typo-brand warnings for the same brand
- Redirect-related matches should be grouped/capped so similar params do not overinflate score

PHONE PIPELINE
File:
- artifacts/api-server/src/lib/pipeline/phone.ts

Implementation goals:
- Normalize once at the top and run all checks against normalized forms
- Add an international format helper that converts 08x -> +3598x for prefix matching
- Separate valid BG mobile logic by named operator groups

Suggested helpers:
- normalizePhone(input)
- toInternationalFormat(input)
- detectBgMobileOperatorPrefix(input)
- isValidBgLandline(input)
- isSyntheticPhonePattern(input)

Risk signals to add:
1. US toll-free / premium style prefixes
   - +1800, +1888, +1900
   - weight: 0.4

2. BG premium SMS short code
   - 5 digits starting with 18
   - weight: 0.45

3. Synthetic repeated / sequential digits
   - e.g. 111111111, 123456789
   - weight: 0.6

4. Unexpected foreign country code
   - not BG (+359), UK (+44), US (+1), DE (+49)
   - mild suspicion only
   - weight: 0.35

5. Invalid Bulgarian international number
   - starts with +359 but invalid mobile/landline/operator prefix
   - weight: 0.5

6. Caller ID spoofing indicator
   - exactly 0000000000 or 9999999999
   - weight: 0.8

7. Malformed or incomplete format
   - structurally odd or invalid length
   - moderate risk

Safe signals to add:
1. Valid A1 mobile
   - 087 / 088
   - weight: -0.15

2. Valid Yettel / Telenor mobile
   - 089
   - weight: -0.15

3. Valid Vivacom mobile
   - 098 / 099
   - weight: -0.15

4. Valid Bulgarian landline
   - +359 2 / +359 3x / +359 4x / +359 5x
   - weight: -0.15

False-positive rules:
- Do not treat every foreign code as highly suspicious
- Safe landline should trigger only for structurally valid landline ranges
- Safe mobile checks should run after normalization and before invalid-prefix inflation

MESSAGE PIPELINE
File:
- artifacts/api-server/src/lib/pipeline/message.ts

Implementation goals:
- Keep existing scam phrase coverage
- Add structured feature extraction rather than only pattern matching

Suggested helpers:
- normalizeMessage(input)

- containsCardLikeNumber(input)
- containsPasswordRequest(input)
- containsCredentialRequest(input)
- containsEmbeddedUrl(input)
- getCapsRatio(input)
- countEmoji(input)
- isTrackingOnlyMessage(input)

Extract features such as:
- contains URL
- URL count
- IBAN presence
- card-number-like presence
- uppercase ratio
- emoji count
- credential request
- institution impersonation
- urgency wording

Risk signals to add:
1. IBAN or card-like number
   - BG IBAN or 16-digit card-like number
   - weight: 0.7

2. Request for personal ID / document data
   - ЕГН, лична карта, личен номер
   - weight: 0.65

3. Request for CVV / CVC / PIN
   - CVV, CVC, ПИН, пин код
   - weight: 0.8

4. Cryptocurrency request
   - Bitcoin, BTC, Ethereum, крипто, портфейл, wallet
   - weight: 0.7

5. Gift card / voucher demand
   - gift card, ваучер, код за, iTunes, Amazon voucher
   - weight: 0.65

6. English phishing phrases
   - click here, verify your account, suspended, unauthorized access
   - weight: 0.6

7. Excessive CAPS ratio
   - more than 40% uppercase letters
   - weight: 0.3

8. Emoji spam
   - 5 or more emojis
   - weight: 0.2

9. Embedded URL
   - contains http:// or https://
   - weight: 0.25

10. Password request
   - парола, password, pwd
   - weight: 0.75

11. Very short message with a single link
   - under 30 chars and contains URL
   - weight: 0.45

Safe signals to add:
1. Formal greeting + named sender
   - starts with Уважаем / Dear
   - ends with organization-like sender signature
   - weight: -0.2

2. Legitimate tracking-only format
   - looks like BG Post / Econt / Speedy tracking code
   - no link
   - no urgency
   - no credential request
   - weight: -0.15

False-positive rules:
- Card-number detection must be boundary-aware
- IBAN detection should be case-insensitive and boundary-aware
- CAPS ratio must count letters only
- Tracking-only safe signal must not trigger if the message includes a link, urgency, or sensitive data request
- Group credential/payment-related matches to avoid unrealistic score stacking

Risk signals to add:

   - weight: 0.35

2. Emotional superlatives
   - невероятно, шокиращо, скандално, невиждано
   - weight: 0.3

3. False urgency / exclusivity
   - само тук, преди официалното, ексклузивно, преди да изтрие

   - Сорос, дълбока държава, НАТОвска агресия, Брюксел командва
   - weight: 0.5

5. Health misinformation markers
   - лекува рак, чудодейно, забранено от фарм, природно лечение победи
   - weight: 0.55

6. Economic clickbait
   - Тайната на богатите, банките крият, пасивен доход, работи от вкъщи без опит
   - weight: 0.5

7. Civil unrest / inflammatory mobilisation
   - свалете правителството, на улицата, бунт, въстание
   - weight: 0.45

8. Pro-Kremlin narrative markers

   - weight: 0.65

9. No normal punctuation / all-caps style
   - lacks sentence-ending punctuation or reads like all-caps shouting
   - weight: 0.25

10. Clickbait title formula
   - endings like:
     - …и ето какво се случи
     - Ще се шашнете
     - Не повярвахте

1. Known Bulgarian institution cited in attribution-like context
   - БТА, МВР, НСИ, БНБ, Министерство
   - weight: -0.25

2. Clear attribution language
   - по данни на, според официални
   - weight: -0.2

3. Balanced language indicators
   - от една страна … от друга
   - спорен въпрос
   - различни мнения
   - weight: -0.15

False-positive rules:
- Missing attribution should not trigger for very short text where attribution cannot be judged reliably
- Political markers should not automatically mean misinformation if the text appears neutral or analytical
- Institution safe signal should only apply in attribution-like context, not anywhere in the body
- Sensationalism-related rules should be grouped/capped to avoid overcounting
